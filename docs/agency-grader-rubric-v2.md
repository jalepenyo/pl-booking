## Problem Statement

The original Agency Grader design had 4 scoring categories (Visibility, Trust, Availability, Conversion) organized by data source. These didn't map cleanly to the PolicyLift funnel and created a competing taxonomy — the report had category names fighting funnel stage labels. The checks themselves included filler ("phone number visible on website") that every agency passes and don't drive value.

This revision aligns the scoring directly with the PolicyLift funnel. The funnel IS the report structure. No competing names.

## Premises

1. The PolicyLift funnel is the right organizing structure — agency owners understand it because it maps to their actual business.
2. The grader scores what's measurable from outside. Everything below Capture & Qualify requires internal data we don't have.
3. Every check must be something most agencies will FAIL — otherwise it's filler that doesn't create urgency.
4. The grader is a 30-second teaser, not the full audit. The full SEO report (which already exists) is the upsell.
5. Discovery is intentionally heavy (10 checks) because SEO problems are the most data-rich and create the strongest "oh shit" moments.
6. Point weighting uses provisional equal-weight per check (see Scoring section). Will be validated against real agency data.

Three scored from outside, one teaser/CTA.

### Stage 1: Discovery — "Can they find you?"

10 checks. Data sources: SEMrush API, Lighthouse/PageSpeed API, Firecrawl.

| # | Check | Detection Method | Notes |
|---|-------|-----------------|-------|
| 1 | Ranks for "[city] insurance" keywords | SEMrush — domain_organic endpoint, filter for local insurance terms | Core local SEO signal |
| 2 | Organic keyword count ≥ 50 | SEMrush — organic_keywords field | Below 50 = effectively invisible |
| 3 | Domain Authority ≥ 20 | SEMrush Authority Score | Most small agencies score 10-30 |
| 4 | Page load under 3 seconds | Lighthouse/PageSpeed Insights API | |
| 5 | Mobile-responsive site | Lighthouse API — mobile score ≥ 70 | |
| 6 | Title/meta includes city + "insurance" | Firecrawl — check title and meta description | Basic local SEO hygiene |
| 7 | Non-brand traffic > 20% | SEMrush — branded keywords identified by exact/fuzzy match on agency name and domain name. Non-brand = all other keywords. Compare traffic from non-brand vs total. | The "oh shit" stat: "95% of your traffic is people who already know your name" |
| 8 | Has dedicated local/coverage pages | Firecrawl — detect pages with URL paths or headings matching coverage types (auto, home, commercial, umbrella) or city/town names beyond primary location. A site with only homepage + about + contact = FAIL. | "269 of 299 keywords point to one page" |
| 9 | Backlink health | SEMrush — toxic vs legitimate backlink ratio. FAIL if toxic backlinks > 5× legitimate count. | Toxic links suppress all rankings |
| 10 | Monthly organic traffic ≥ 200 | SEMrush — monthly traffic estimate. Threshold: 200 visitors/month [TO BE VALIDATED against real agency data]. | Below 200 = negligible organic presence |

Maps to: SEO & AEO Optimized Website, Digital Ads Automation, Website Wizard

CTA tie-in: "Want the full SEO audit? Book a demo." → The existing PolicyLift SEO report is the upsell from this stage.

**Note:** This stage is intentionally heavy. SEO problems produce the most concrete, personalized "oh shit" data points. "You rank for 299 keywords but only 3 people a month are actually shopping for insurance" is a gut punch.

**Simplification for later:** Some of these checks may be collapsed or removed based on API cost and detection reliability. The 10 are the candidate set — final count may be fewer.

### Stage 2: Trust — "Do they trust you?"

5 checks. Data sources: Google Places API, Firecrawl.

| # | Check | Detection Method | Confidence | Notes |
|---|-------|-----------------|------------|-------|
| 1 | Google rating ≥ 4.5 | Places API — rating field | High | |
| 2 | 50+ Google reviews | Places API — user_ratings_total | High | |
| 3 | Review within last 30 days | Places API — check review timestamps | Low | API returns 5 curated reviews, not necessarily most recent. False negatives possible. Worth keeping: most agencies genuinely don't have recent reviews, so when verifiable this is a strong signal. Mark "Unable to verify" if no timestamps available. |
| 4 | Owner responds to reviews | Places API — check for reply field on reviews | Low | Same API caveat. Most agencies don't reply to reviews — when detectable, this differentiates. Mark "Unable to verify" if review sample too small. |
| 5 | Reviews showcased on website | Firecrawl — detect embedded review widgets, testimonial sections, or known review platform scripts (Birdeye, Podium, Grade.us, etc.) | Medium | Are they leveraging social proof on their own site? |

Maps to: Google Review Automation, Reputation Management

### Stage 3: Capture & Qualify — "Can they reach you and start the process?"

4 checks. Data sources: Google Places API, Firecrawl.

| # | Check | Detection Method | Confidence | Notes |
|---|-------|-----------------|------------|-------|
| 1 | After-hours availability | Two detection paths (either = PASS): (a) Firecrawl detects voice AI scripts, after-hours chatbot, or "24/7"/"after hours" language near contact info — this is the real signal; (b) Places API hours show coverage past 9pm or before 8am on any day, OR any weekend hours — baseline signal. Path (a) alone is enough. Path (b) alone counts as partial. Neither = FAIL. | Medium | The "10pm Saturday fender bender" test. Not labeled "24/7" since most won't truly be 24/7 — but any after-hours solution counts. |
| 2 | AI-powered chat | Firecrawl — detect known AI chat platforms: Ada, Drift AI, Intercom Fin, Tidio AI, Salesforce Einstein, PolicyLift Chat AI. Generic live chat widgets (Tawk.to, basic LiveChat, Olark) = FAIL — they require staff online. Unknown widget = "Unable to verify." | Medium | Maintain platform classification in `src/lib/ai-chat-platforms.json` in the repo. Updated as new platforms found during QA testing. |
| 3 | Insurance-specific forms | Firecrawl — detect forms that ask coverage type, vehicle, property, etc. Match ≥2 insurance terms. | Medium | A "name, email, message" contact form doesn't count. |
| 4 | Self-service booking/calendar | Firecrawl — detect Calendly, Acuity, HubSpot meetings, or booking-related links | Medium | Can a prospect schedule time without calling? |

Maps to: Voice AI (Kelly), Chat AI, Smart Forms, Booking Tools, 2-Way Texting

## Scoring

**Input:** Google Business Profile (selected via Places Autocomplete). From the GBP we extract the domain URL for all website-dependent checks.

**Total checks:** 19 across 3 scored stages. **100 points total.**

**Provisional point allocation [TO BE VALIDATED]:** Equal weight per check within each stage. Stages weighted to 100 total:
- Discovery (10 checks): 50 pts → 5 pts per check
- Trust (5 checks): 25 pts → 5 pts per check
- Capture & Qualify (4 checks): 25 pts → ~6 pts per check (6+6+6+7)

Discovery is intentionally weighted heavier (50%) because SEO problems are the most actionable and produce the strongest upsell into the full SEO audit. Point values will be recalculated once the final check list for Stage 1 is confirmed (some checks may be collapsed — see Simplification note under Stage 1).

**Letter grades:**

| Grade | Score | Meaning |
|-------|-------|---------|
| A | 80-100 | Strong digital presence |
| B | 65-79 | Good foundation — notable gaps |
| C | 50-64 | Average — significant opportunities |
| D | 35-49 | Below average — major gaps |
| F | 0-34 | Critical — needs immediate attention |

**Confidence handling:** If a check can't be verified, display "Unable to verify" with neutral icon. Do NOT score as failure. Score formula: `(points earned on verified checks / total points of verified checks) × 100`, then apply grade thresholds. Display as: "45*/100 — *3 checks could not be verified and are not counted against you." Confidence levels (High/Medium/Low) are informational for future calibration — they do not change runtime scoring behavior.

**Input validation:** If Places Autocomplete returns zero results, show: "We couldn't find [input] as a business. Try the exact name as it appears on Google, including city and state." Offer text search fallback. If selected GBP has a `website` field pointing to a non-agency domain (facebook.com, yelp.com, etc.) or no website at all, treat as "no website found" and trigger the no-website flow.

**Failure mode definitions:**
- API returns error (429, 500, timeout) → mark affected checks as "Unable to verify"
- SEMrush returns no data for domain (brand new or unindexed site) → checks 1, 2, 7, 10 score as FAIL (no presence = discovery failure); check 3 = "Unable to verify"
- Firecrawl times out (15s limit) → mark affected checks as "Unable to verify"
- No website found at all → skip all website-dependent checks; score only Trust stage from Places data; show "We couldn't find a website" as strongest CTA for Website Wizard

**Scan timing:** "30 seconds" refers to user-perceived time. All API calls fire in parallel. Results stream via SSE as they arrive. Animation layer presents results on a choreographed timeline. If all APIs respond in 5 seconds, animation still runs ~30 seconds for the emotional reveal. 60-second hard timeout — show whatever is ready, mark the rest "Unable to verify."

**Caching:** SEMrush data cached 7 days per domain (SEO changes slowly). Places data cached 24 hours per Place ID. Firecrawl/Lighthouse cached 24 hours per domain.

**Firecrawl call strategy:** A single Firecrawl request per domain retrieves HTML, title, meta, and visible text. All Firecrawl-dependent checks (Stage 1: checks 6, 8; Stage 2: check 5; Stage 3: checks 1b, 2, 3, 4) are evaluated against that single response to minimize cost and latency.

**SEMrush non-brand traffic (Check 7):** Cap keyword retrieval at top 100 by traffic volume and apply brand detection to that sample. A keyword is branded if it contains the agency's business name (exact substring, case-insensitive) or the root domain (strip TLD). All other keywords are non-brand. This is a sampling approach — note in the report if sample size is small.

**SEMrush backlinks (Check 9):** Requires SEMrush Backlink Audit API access. Verify API plan includes this endpoint before building. If unavailable, mark check as "Unable to verify."

**Animation/SSE handoff:** Results are revealed in animation-order as the timer passes each check's scheduled reveal time. Early-arriving API results are buffered and displayed when their animation slot is reached. Late results show a spinner at their slot until they arrive or timeout.

## What Changed from the Original Design

| Aspect | Original | Revised |
|--------|----------|---------|
| Scoring structure | 4 categories by data source (Visibility, Trust, Availability, Conversion) | 3 funnel stages + 1 CTA (Discovery, Trust, Capture & Qualify, Intake→Service) |
| Funnel relationship | Funnel was a separate narrative wrapper | Funnel IS the report structure |
| Check count | 17 checks | 19 checks (added non-brand traffic, local pages, backlink health, traffic volume, reviews on website; removed phone number visible, generic contact channels) |
| Availability concept | "Extended hours beyond 9-5" (catching 8:30am openings) | Reframed as "24/7 coverage" — voice AI, after-hours chat, or weekend/evening hours that actually matter |
| Lead Capture concept | Generic forms and channels | Insurance-specific forms, AI chat (not just any widget), self-service booking |
| Filler checks | Phone number visible, multiple contact channels | Removed — every check should be something most agencies fail |
| SEO tie-in | Standalone category | Ties to existing PolicyLift SEO audit report as upsell |
| Teaser stages | 3 separate stages (Intake, Quote & Bind, Service Renew Grow) | 1 combined CTA stage (Intake → Quote → Bind → Service) |

## Open Questions

1. **Discovery simplification** — 10 checks may be too many for a 30-second scan. Which can be collapsed once we test against real agencies?
2. **Traffic threshold calibration** — 200 visitors/month is provisional. Needs validation against 20+ real agency sites to find the right cutoff.
3. **SEO report upsell flow** — how does the grader CTA connect to the existing full SEO audit? Same demo booking, or separate "Get your full SEO report" CTA?
4. **SEMrush API plan** — verify current plan includes Backlink Audit API access for Check 9. If not, either upgrade or drop that check.

## What I noticed about how you think

- You kept pushing back when the categories felt "marketing-y" vs technical. "I'm really wed to the original funnel, it just sounds so accurate and technical." You have an instinct for when language is for the customer vs for the pitch deck — and you want the grader to feel like the former.
- You caught that "extended hours" was measuring the wrong thing before I did. "It's more about evenings and weekends vs our thing will count 8:30am." That's the difference between a metric that looks good on paper and one that actually matters.
- When I proposed filler checks like "phone number visible on website," you called it immediately: "this is stupid, we are grasping for straws." You'd rather have 4 checks that matter than 8 that pad the report.
- You instinctively collapsed the teaser stages from 3 to 1 — "no point in 2 stages for teaser, just that we also solve the bottom half with AI." Simplification over structure for structure's sake.
